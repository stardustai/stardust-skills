# Technical Debt Standard

状态：滚动设计稿。

## 1. 定义

技术债是已经存在于代码、架构或运行方式中的结构性问题。它可能暂时不影响当前
运行，但会让后续修改更容易出错、更难测试、更难恢复，或者必须持续通过绕路和
补丁才能开发。

技术债不能仅凭“代码不优雅”或个人偏好认定。每项债务必须有代码、配置、测试、
运行或历史变更证据。

## 2. 判定条件

至少满足一项：

- 局部修改会产生无法预测的跨模块影响；
- 同一业务能力存在多套事实源或重复实现；
- 模块职责混乱，无法明确变更位置；
- 公共接口、数据结构或状态变化没有稳定合同；
- 实现与已批准系统架构不一致；
- 关键行为无法通过自动测试稳定验证；
- 测试局部功能必须依赖大量 Mock、全局状态或完整生产环境；
- 依赖关系循环、方向错误或跨层调用；
- 错误处理掩盖根因，失败后无法恢复；
- 权限、敏感数据或 Secret 边界不清；
- 运行时约束因系统结构而无法满足；
- 已知临时方案长期存在，新功能必须继续适配；
- 关键故障缺少日志、监控或回滚能力。

## 3. 常见类型

- 架构债：组件越层调用、多个模块拥有同一状态。
- 职责债：单一文件或模块同时承担接口、业务、存储和通知。
- 接口债：调用者依赖内部字段，输入输出没有稳定合同。
- 数据债：同一字段含义不一致，迁移和事实源不清。
- 测试债：核心业务无法隔离验证，测试依赖真实生产环境。
- 依赖债：循环依赖、重复库或已产生实际维护风险的废弃依赖。
- 安全债：权限检查可绕过，敏感数据可能进入日志。
- 运行债：缺少健康检查、告警、恢复或可靠回滚。
- 补丁债：通过硬编码、特殊分支、额外重试或兼容层绕过根因。

## 4. 不自动属于技术债

- 单纯的代码风格和命名偏好；
- AI 不熟悉的实现方式；
- 尚未批准的新功能；
- Spec 本身不完整；
- 边界清楚、可稳定复现的普通 Bug；
- 没有安全、兼容或维护影响的旧依赖；
- 为当前需求合理增加的必要复杂度；
- 只以“可以更优雅”为理由的重构建议。

## 5. 与其他问题的边界

| 问题 | 定义 | 处理方式 |
| --- | --- | --- |
| 产品缺口 | Spec 未说明做什么 | 返回 `spec-intake` |
| 普通 Bug | 实现不符合已批准行为 | 系统化排障和回归测试 |
| 工程准备缺口 | 缺 README、测试入口、Eval、Remote 或 Pre-commit | 初始化补齐 |
| 技术债 | 系统结构使正确修改、测试或恢复不可靠 | 全量或最小干净修复 |
| 新架构需求 | 原架构不能承载新能力 | 更新 Technical Design 并确认 |
| 过程违规 | Agent 未遵守沟通、证据或审批流程 | 阻断当前步骤并纠正流程 |

## 6. AGENTS 规则

适用的 `AGENTS.md` / `AGENT.md` 中已经固化为代码或架构结果的违规属于技术债，
包括但不限于补丁式逻辑、以兼容或兜底掩盖旧设计、硬编码业务判断、无必要复杂度、
缺少相应测试，以及不可逆变更缺少安全恢复设计。

缺文档、测试入口、Remote 或 Pre-commit 属于工程准备缺口；Agent 沟通和执行方式
违规属于过程违规。

Graphify 不属于公司通用规则，缺少或未更新 Graphify 不构成技术债。

## 7. 项目登记格式

每项技术债至少记录：

```text
TD-ID
问题描述
规则来源（如适用）
代码或运行证据
影响模块
为什么增加后续风险
是否被本轮功能触及
全量修复方案
最小干净修复方案
验证方法
负责人
状态
关闭证据
```

不得只写“架构不好”“代码需要优化”或“建议重构”。

## 8. 修复选择

用户只有两个选项：

- 全量修复本轮审计识别的全部技术债；
- 最小干净修复本轮触及和依赖范围内的全部技术债。

不允许选择“保留本轮涉及的技术债并继续打补丁”。

## 9. 关闭条件

技术债只有在以下条件全部满足时才能关闭：

- 根因已经消除，不是新增兼容或补丁；
- 修复后模块职责、接口和依赖边界明确；
- 相关测试先正确失败、修复后通过；
- 完整回归和适用 Eval 通过；
- 架构、测试、Runtime 和追踪文档同步；
- 独立 Reviewer 检查修复没有引入新的债务；
- 关闭证据写入 `technical-debt-register.md`。
