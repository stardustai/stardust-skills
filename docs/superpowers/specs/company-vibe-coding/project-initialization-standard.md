# Existing Project Initialization Standard

状态：滚动设计稿。

## 1. 入口原则

基于已有项目开发时，不得直接增加功能。总控 Skill 必须先补全工程合同、建立真实
运行与测试基线、完成架构审计，并修复本轮不能接受的技术债。

## 2. 初始化顺序

1. 读取仓库和目录范围内适用的 `AGENTS.md` / `AGENT.md`。
2. 检查 Git 状态、Remote URL、默认分支、当前分支和基线 Commit。
3. 读取 README、架构、依赖、构建、测试、部署和运行约定。
4. 找到与本轮功能相似的现有实现、公共组件、数据模型和接口。
5. 从干净环境尝试安装、构建、启动、健康检查和现有测试。
6. 按职责盘点已有文档和缺失内容，不按文件名或固定目录直接判定缺失。
7. 结构不符合推荐模板时，让用户二选一：改成标准结构，或就地适配结构。
8. 按选择补齐 README、PROJECT.yaml 和适用内容；就地适配时在两者中记录真实路径，
   缺失内容放在最合适的既有位置，不创建重复事实源。
9. 由独立 Review Agent 按真实路径、SHA-256、必需主题和精确定位文本验证内容齐全；
   缺失时由 AI 补齐并重新审查。
10. 建立当前功能、测试、Eval 和运行基线，明确无法验证的部分。
11. 对照 Spec 检查范围、行为、数据、权限、架构和运行差异。
12. 生成技术债登记和独立架构修复计划。
13. 让用户选择全量修复技术债或最小干净修复。
14. 完成架构修复、完整验证和独立 Commit。
15. 记录新的初始化基线和 `last_green_commit` 后，才进入功能计划。

文档组织选择与技术债修复选择相互独立，均需记录。前者决定资料放在哪里，后者决定
哪些结构性问题必须修复。

## 3. 输出事实分类

初始化报告必须区分：

- 由代码、配置或运行证明的事实；
- 有证据但仍需确认的推断；
- 缺失文档和工程准备缺口；
- 普通 Bug；
- 产品或 Spec 缺口；
- 技术债；
- 新架构需求；
- 无法运行、测试或 Eval 的阻断项。

## 4. 用户只获得两个技术债处理选项

### A. 全量修复技术债

关闭本轮系统化审计识别的全部技术债，然后建立新的稳定基线。

### B. 最小干净修复

缩小修复范围，但必须修复：

- 本轮功能将修改的模块；
- 本轮功能直接或间接依赖的既有模块；
- 影响本轮正确性、测试性、权限、数据或运行稳定性的问题；
- 为本轮建立清晰接口和职责边界所必需的问题。

本轮功能不得引用、扩展或绕过未修复的坏架构。只要触及已知债务，该债务自动进入
修复范围。最小干净修复不能增加兼容层、特殊分支或临时补丁。

如果最小干净修复仍不能形成清晰、可测试边界，只能改为全量修复或缩小/取消功能。

## 5. 修复执行

- 架构修复必须使用独立 Design、Plan、任务和 Commit；
- 修复前建立现有行为特征测试，防止静默改变未知合同；
- 按模块和依赖顺序小批修复；
- 每批运行完整测试并进行独立 Review；
- 每批更新架构和技术债文档；
- 全部修复后运行完整测试、Eval 和 Runtime 验证；
- 功能开发不得与初始化修复混在同一 Commit。

## 6. 初始化完成门禁

只有以下条件全部成立，才能开始功能开发：

- 仓库合同完整并通过校验；
- 标准结构或就地适配选择已记录，README 与 PROJECT.yaml 的文档导航一致；
- 全部映射文档已通过绑定当前 Hash 与精确定位的独立内容审查；
- Remote、分支和基线明确；
- 项目可以按合同安装、构建、运行和测试；
- 当前 Spec 与代码差异已经处理；
- 用户选择的技术债修复范围全部关闭；
- 架构文档与修复后代码一致；
- 完整测试和适用 Eval 通过；
- 新的初始化基线 Commit 和 `last_green_commit` 已记录。
