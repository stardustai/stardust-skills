# 招聘群面评同步

在小青正式面评提交成功后执行本流程。目标是让招聘团队和前轮面试官及时看到最终判断，并明确知道后轮为什么维持、提高或推翻了前轮结论。

## 触发和边界

- 只在 `upload_interview_result(dry_run=false)` 成功后发送；草稿、讨论、听记初步分析和 dry run 不发送。
- 用户对最终面评明确说“提交”，即同时授权发送标准招聘群同步消息；用户明确说“不发群”“只提交小青”时跳过。
- 小青正式提交成功、群发送失败时，不回滚小青结果。分别报告两个动作的状态。
- 不发送完整 Transcript、简历原文、薪资细节或敏感个人信息。只发送招聘决策所需的结论、证据摘要、风险和小青内部链接。

## 解析接收对象

1. 从 `list_candidate_interviews` 读取当前轮次之前的全部正式面试记录，提取去重后的面试官姓名。不要 @ 当前轮面试官本人，除非他也参加了更早轮次。
2. 读取 `dingtalk-chat`、`dingtalk-contact` 和 `dws-shared` skill，使用当前 DWS CLI 能力解析群和人员。
3. 群定位优先级：
   - 小青、日历、历史消息或候选人资料中已经给出的招聘群；
   - 与岗位族直接对应的精确招聘群，例如 `【招聘】售前`；
   - 使用 `dws chat search --query "<连续群名关键词>" --format json` 搜索得到的唯一精确匹配群。
4. 不要仅凭候选人姓名创建或猜测群名。搜索返回多个合理群时停止发送，并列出候选群等待用户确认。
5. 对每位前轮面试官，先用通讯录精确解析其 `openDingTalkId`。同名、查不到或不在目标群时，不伪造纯文本 @；报告该人员无法解析。其他已确认人员仍可正常 @。

## 判断差异

逐轮比较前轮与本轮的：

- 推荐结论和总分；
- 候选人适配的岗位级别；
- 前轮主要正向判断；
- 本轮新增的直接证据；
- 哪个前轮假设被证实、削弱或推翻；
- 为什么差异会改变推荐、定级或薪资判断。

不要写成“前轮面试官判断错误”。应写成证据增量，例如：

```text
与一面差异：一面基于履历和项目描述判断为弱推荐 74；二面现场核验代码贡献边界、评测体系和人才管理后，发现原先对 AI 工程深度、动手能力和团队管理的假设未被证明，因此调整为 L5 不推荐 52。若按 L4 平台架构岗位，可重新核验后考虑。
```

当结论一致时也要说明新增证据，不要只写“与前轮一致”。

## 筛选建议

根据本轮新增证据，给 HR、简历筛选人和前轮面试官留下 2-4 条可执行的筛选纠偏。必须回答两个问题：

1. **以后应排查什么**：哪些包装、错配或风险应在简历筛选、HR 面或前轮面试中更早暴露。
2. **以后应加强什么**：哪些关键能力证据验证不够，需要增加追问、现场题、代码/作品核验或背景交叉验证。

每条建议都要包含“触发信号 + 验证动作 + 判断标准”，并与本候选人的实际证据对应。不要写“加强技术考察”“关注沟通能力”之类无法执行的通用话。

示例：

```text
- 排查开源贡献包装：候选人用 commit 数量证明技术影响力时，前轮必须核验仓库创建时间、原创代码占比、核心设计提交和 upstream 关系；若主要是 fork、批量导入或一天搭建，不得作为长期技术深度证据。
- 加强高级岗位管理验证：不能只看团队规模，应要求候选人讲一次招强人、处理低绩效、组织拆分或跨部门冲突，回答必须包含本人决策、反对意见、结果指标和复盘；只有方法论而无真实案例，管理能力不得高分。
```

建议应聚焦可迁移的筛选规则，不重复本轮面评结论。若本轮没有足够证据形成可靠纠偏，明确写“本轮无新增筛选规则”，不要臆造。

## 消息格式

标题：`面评同步｜<候选人>｜<岗位>｜<轮次>｜<结论>`

正文保持紧凑：

```markdown
<真实@占位符>

**本轮结论**：<推荐结论>，<总分>；<定级判断>。

**核心判断**：
- <最重要的能力判断>
- <最重要的风险或红线>
- <是否建议继续及适合级别>

**与前轮的判断差异**：
- <前轮面试官/轮次/结论/分数>
- <本轮新增证据>
- <被证实或推翻的假设，以及结论变化原因>

**后续筛选建议**：
- 排查：<触发信号 + 验证动作 + 判断标准>
- 加强：<当前证据缺口 + 下一轮应增加的验证方式 + 通过标准>

**面评链接**：<feedback_url>
```

有多位前轮面试官时，按轮次分别写差异；不要把不同面试官的判断合并成一个模糊结论。

## 发送和验证

1. 使用 `dws chat message send`，通过 `--at-open-dingtalk-ids` 传入去重后的真实 ID，正文必须包含对应 `<@openDingTalkId>` 占位符。
2. 使用稳定的 `--uuid` 防止同一次小青提交重复发送。建议：`interview-feedback-<interview_id>-<revision_id>`；将 revision 中不适合 UUID 的字符规范化。
3. 发送前核对群名、群 ID、@对象、候选人、岗位、轮次、结论、分数和链接均来自已确认事实。
4. 发送成功后记录群名、@人员和消息返回标识。必要时用群消息查询复查，但不要通过重复发送测试。
5. 如果面评随后被用户修改并重新正式提交，发送一条标题含“面评更正”的新消息，说明旧结论、新结论和改变依据；不要静默覆盖，也不要让群里同时存在两条看似都有效的结论而无解释。

## 失败状态

- `skipped`：用户明确要求不发送，或没有前轮面试官且用户只要求 @ 前轮面试官。
- `blocked`：群不唯一、人员同名、无法解析真实 @、目标群不可访问。
- `error`：发送接口或认证失败。

失败时给出可执行的下一步，但不要改发到个人、其他招聘群或使用纯文本姓名冒充 @。
