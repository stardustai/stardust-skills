# 输出格式规范

## 固定表头

最终测试用例必须使用 7 列，顺序不可调整：

| 用例名称 | 所属模块 | 前置条件 | 步骤描述 | 预期结果 | 用例等级 | 编辑模式 |
| --- | --- | --- | --- | --- | --- | --- |

## 字段规则

| 字段 | 规则 |
| --- | --- |
| 用例名称 | 唯一且表意清晰，命名建议为 `[场景]-[类型]`，禁止“创建成功”这类模糊名称 |
| 所属模块 | 固定展示 `/<需求名称>`，同一份需求全部用例必须完全一致 |
| 前置条件 | 可执行、可验证，禁止“正常操作”“适当准备”等空泛描述 |
| 步骤描述 | 使用 `[1]...；[2]...` 编号化描述 |
| 预期结果 | 使用与步骤一致的 `[1]...；[2]...` 编号，断言要明确 |
| 用例等级 | `P0` 核心流程，`P1` 重要功能，`P2` 一般补充，`P3` UI展示样式和文字描述错误 |
| 编辑模式 | 固定 `STEP` |

## 内容限制

1. 预期结果中的编号不能多于步骤描述中的编号。
2. 一个操作步骤需要校验多个结果时，在同一个 `[n]` 下用 `；` 分隔。
3. 不要额外生成一条只验证同一步其他断言的重复用例。
4. 所有单元格文案内部不要包含 literal `\n`、`<br>`、`<br/>`。
5. Markdown 表格中如出现 `|` 或 `\`，必须转义；生成 CSV/XLSX 时脚本会处理真实单元格。
6. 正式 Excel 输出不能使用历史的 10 列内部评审格式。

## 输出顺序

1. 先输出测试用例表格。
2. 再输出待确认问题。
3. 最后输出覆盖说明，按功能点、异常场景、边界值、角色权限、流程覆盖总结。
