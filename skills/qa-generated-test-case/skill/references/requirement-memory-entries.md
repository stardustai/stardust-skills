# 业务记忆条目：算法池质检失败处理与上游报错联动

## 1. 基本信息

- 记忆标题：算法池质检失败处理与上游报错联动
- 来源需求：
  1. 2026.04.30《算法池支持退回质检算法未通过数据》
  2. 2026.05.08《标注页面新增 上游报错 功能》
- 需求日期：2026-05-11 整理入库
- 所属模块：算法池质检配置 / 算法池质检失败处理 / 上游报错展示
- 关键词：算法池、质检失败、滞留算法池、驳回原操作人、报错状态、上游报错

## 2. 这条需求稳定下来的业务规则

1. 只有在算法池配置了质检算法时，才展示“质检未通过数据处理方式”配置项。
2. 未配置质检算法，或配置后又清空质检算法时，该配置项都应隐藏。
3. 失败处理方式默认值为“滞留算法池（Staging algorithm pool）”。
4. 失败处理方式至少包含“滞留算法池”和“驳回至原操作人（Return to original operator）”。
5. 质检失败且选择“滞留算法池”时，任务留在算法池并进入“报错”状态。
6. 质检失败且选择“驳回至原操作人”时，系统尝试退回原操作人。
7. 退回原操作人失败时，任务回退为“滞留算法池 + 报错状态”逻辑。
8. 本需求范围内默认质检失败场景都会返回 message，不考虑空 message 分支。
9. 质检失败返回的每条 message 都应写入独立“上游报错”记录，不需要写入“报错”详情。
10. 算法池类上游报错标题固定为：`该任务在上游算法池质检未通过被驳回，报错信息如下：`
11. 产品功能只要求任务流转后必须能展示上游报错，具体落库时机不作为本轮产品规则约束。
12. 算法池中处于“报错”状态的任务，不再执行任何算法；手动触发与自动触发都应跳过。

## 3. 角色 / 状态 / 提交限制

- 角色：
  1. 项目经理：配置算法池质检失败处理方式；对报错任务执行错误处理。
  2. 原操作人：接收退回任务；查看算法池上游报错。
  3. 下游工作池执行人：接收错误处理流转的任务；查看算法池上游报错。
- 关键状态：
  1. 算法池正常执行中
  2. 算法池报错状态
  3. 原操作人收到退回任务
  4. 下一个工作池收到带算法池上游报错的任务
- 限制：
  1. 算法池报错状态任务不能再跑任何算法。
  2. 算法池上游报错仅作为背景提示，不阻断下一站处理。

## 4. 上下游依赖

1. 依赖算法池质检算法返回通过/失败和 message。
2. 依赖原操作人退回能力判断是否成功。
3. 依赖错误处理流转能力，把算法池报错任务送往其他工作池。
4. 依赖上游报错存储与读取能力，展示算法池失败原因。
5. 依赖“上游报错只展示一站”的通用生命周期规则。

## 5. 容易漏测的点

1. 未配置质检算法时却错误展示失败处理方式。
2. 质检算法被清空后，失败处理方式没有同步隐藏。
3. 退回原操作人失败时，没有回落到算法池报错状态。
4. 多条失败 message 没有逐条生成多条上游报错记录。
5. 失败 message 被错误写到“报错”而不是“上游报错”。
6. 算法池报错状态任务仍被手动或自动触发重新跑算法。
7. 项目经理错误处理流转后，下一站未看到算法池上游报错。

## 6. 和历史需求的关系

- 覆盖了哪些旧规则：
  1. 原先算法池质检失败后只能滞留且无法继续处理的旧逻辑。
- 修改了哪些旧规则：
  1. 算法池质检失败后，新增由项目经理预设处理方式的能力。
- 新增了哪些规则：
  1. 失败处理方式配置项。
  2. 驳回至原操作人。
  3. 退回失败兜底。
  4. 报错状态不再重跑算法。

## 7. 后续新需求生成用例时应该额外提醒模型关注什么

1. 任何涉及算法池质检失败后任务去向的新需求，都要优先检查是否影响“滞留算法池 / 驳回原操作人 / 失败兜底”三条路径。
2. 任何涉及算法池报错任务再次触发的需求，都要优先回归“报错状态任务不再跑算法”的规则。
3. 任何涉及算法池失败 message 展示的新需求，都要同时检查“报错”和“上游报错”的边界是否被破坏。

# 业务记忆条目：强校验 / 弱校验与上游报错联动

## 1. 基本信息

- 记忆标题：强校验 / 弱校验配置与上游报错联动
- 来源需求：
  1. 2026.05.07《标注池、检查池、抽查池配置质检算法支持选择强校验与弱校验（更新版）》
  2. 2026.05.08《标注页面新增 上游报错 功能》
- 需求日期：2026-05-11 整理入库
- 所属模块：工作流配置 / 当前工作池任务质检 / 下游工作池上游报错展示
- 关键词：强校验、弱校验、上游报错、工作池流转、多记录、生命周期

## 2. 这条需求稳定下来的业务规则

1. 新增质检算法时，校验类型默认选择强校验。
2. 历史未配置校验类型的算法，当前回显规则按强校验处理。
3. 当前池任务完成后，无论配置多少个质检算法，都要并行调用全部算法。
4. 当前池只要有任一强校验失败，任务就不可提交。
5. 当前池若失败算法仅包含弱校验，任务可提交。
6. 当前池弱校验失败直提时，需要把弱校验失败信息写入上游报错。
7. 多个弱校验算法同时失败时，需要写成多条上游报错记录。
8. 上游报错只在下一个工作池展示一站。
9. 上游报错详情过长时截断，支持特殊字符。
10. 上游报错写入失败不影响任务流转，但属于缺陷。

## 3. 角色 / 状态 / 提交限制

- 角色：
  1. 项目经理：配置质检算法强弱类型。
  2. 当前工作池执行人：开始标注、完成、查看当前池报错、提交允许提交的任务。
  3. 下游工作池执行人：查看上游报错。
- 关键状态：
  1. 等待提交 + 校验中
  2. 等待提交 + 校验通过
  3. 等待提交 + 强校验未通过
  4. 等待提交 + 弱校验未通过
- 提交限制：
  1. 强校验未通过不可提交。
  2. 弱校验未通过可提交。

## 4. 上下游依赖

1. 依赖质检算法返回通过 / 失败和 message。
2. 依赖提交链路支持在弱校验失败直提时写入上游报错。
3. 依赖下一个工作池读取并展示上游报错。
4. 依赖任务流转引擎控制“只展示一站”的生命周期。

## 5. 容易漏测的点

1. 混合强弱场景下的优先级。
2. 多个弱校验失败后，上游报错记录数是否正确。
3. 强校验失败是否被错误放行。
4. 弱校验失败是否被错误阻塞。
5. 上游报错是否在后续工作池残留。
6. 当前池返工后通过时，旧弱校验信息是否错误沿用。

## 6. 和历史需求的关系

- 覆盖了哪些旧规则：
  1. 原先“所有质检算法都按强校验处理”的旧逻辑。
- 修改了哪些旧规则：
  1. 当前池失败后不再一律阻塞提交，而是区分强校验和弱校验。
- 新增了哪些规则：
  1. 上游报错。
  2. 弱校验失败直提写入上游报错。
  3. 上游报错多记录和一站生命周期。

## 7. 后续新需求生成用例时应该额外提醒模型关注什么

1. 任何涉及任务提交、跨池流转、报错透传的新需求，都要先检查是否影响强弱校验和上游报错规则。
2. 任何涉及当前池状态文案、任务可提交性的新需求，都要优先回归强弱混合场景。
3. 任何涉及 message 展示的新需求，都要同时考虑当前池报错和下游上游报错是否一致。

# 业务记忆条目：项目进度统计

## 1. 基本信息

- 记忆标题：项目进度统计
- 来源需求：2026.05.07《项目进度统计；具体指标：整体进度，已交付量，已用成本，验收通过率》
- 需求日期：2026-05-12 整理入库
- 所属模块：项目统计 / 项目进度统计 / 项目创建编辑
- 关键词：整体进度、已交付量、已交付帧量、已用成本、验收通过率、计划交付总量、抽查池

## 2. 稳定业务规则

1. 项目统计下新增“项目进度统计（Project Progress Overview）”页面。
2. 页面统计维度是项目维度，表格仅展示当前项目一行。
3. 时间范围默认项目启动当天到系统当天，用户选择起止时间并确认后刷新统计结果。
4. 时间范围不可早于项目启动日，不可晚于系统当天，截止时间必须大于等于起始时间。
5. 整体进度 = 已交付量（任务）/ 计划交付总量，百分比保留 1 位小数，允许超过 100%。
6. 计划交付总量在创建/编辑项目页配置，非必填，仅支持整数，不能为 0。
7. 计划交付总量未填写时，整体进度展示“未填计划交付总量”。
8. 已交付量（任务）统计所选时间范围内首次提交到完成池的数据量，不重复计数。
9. 已交付量（帧）统计完成池数据中的有效帧量，不包含无效帧。
10. 数据到达完成池后，即使后续被驳回再进入完成池，也只按首次提交时间统计一次。
11. 已用成本 = 所选时间范围内所有供应商产量 * 标注成本的求和。
12. 操作项类型价格未配置时，对应产值按 0 计算；项目整体无可用价格配置时显示 0。
13. 私有化环境项目中，已用成本字段空缺，其余统计字段正常显示。
14. 验收通过率按抽查池粒度统计，总通过率、首轮通过率、复验通过率都受页面时间范围控制。
15. 验收总量只包含抽查池中被处理为“通过”或“驳回”的数据。
16. 通过率分母为 0 时，页面展示“无数据”。
17. 验收通过率表格单元格只展示“查看详情”，具体数据在详情弹窗中查看。
18. 导出复用已有导出能力，确认后按当前所选时间范围导出 Excel。
19. 导出文件第一行展示当前所选时间，第二行展示统计字段。
20. 每个抽查池在导出字段中分别生成总通过率、首轮通过率、复验通过率三列。
21. 页面数据每 2 小时定时刷新一次。
22. 历史数据不兼容，历史项目字段为空，仅对功能上线后创建的项目进行统计。

## 3. 容易漏测的点

1. 已交付量重复统计被驳回后再次进入完成池的数据。
2. 已交付帧量把无效帧计入。
3. 整体进度未处理计划交付总量为空或超过 100% 的场景。
4. 已用成本在价格缺失或私有化项目场景展示错误。
5. 验收通过率没有按处理动作发生时间过滤。
6. 首轮通过率和复验通过率没有区分首次进入抽查池。
7. 多抽查池详情弹窗或导出动态列错位。
8. 历史项目被错误补算。

## 4. 后续生成用例提醒

1. 涉及项目统计的新需求，要优先检查时间范围、统计口径、导出字段和历史兼容。
2. 涉及完成池交付量的新需求，要回归“不重复计数”和“首次提交时间固定”。
3. 涉及验收质量的新需求，要区分总通过率、首轮通过率、复验通过率。
4. 涉及成本的新需求，要同步关注价格缺失、私有化项目和供应商效能统计依赖。

## 3. Rosetta：标注生产主链路

1. Rosetta 的核心业务链路是：项目配置 → 导入数据集或数据下发 → 工作流/工作池执行 → 标注池 → 检查池 → 抽查池 → 完成池。
2. 人工标注页面可进入工作详情页；工作详情页展示项目配置好的工作池。
3. 标注池、检查池、抽查池均可从人工标注页或工作详情页进入；若工作池无剩余任务，需要提示“当前工作池剩余任务为0，无法分配”。
4. 标注池点击完成并提交后，数据进入检查池，且实例保留标注池标注结果。
5. 检查池点击合格后，数据进入抽查池，且实例保留检查池标注结果。
6. 抽查池点击合格后，数据进入完成池，且实例保留抽查池标注结果。
7. 检查池和抽查池支持点击修改后继续标注；检查池修改后数据在检查池展示，抽查池修改后数据在完成池和抽检包列表查看结果中展示。
8. 检查池可点击退回，并标记错标；退回到上一操作员的数据只有上一操作员可见，退回到原操作员的数据只有原操作员可见。
9. 抽查池不合格数据可标记错标并驳回；抽检包任务处理可将不合格任务驳回到上一操作员或原操作员。
10. 完成池需要展示被标记的错标信息。
11. 错标存在状态流转，例如开启、待验收、自检无错误、已验收、未解决；错误状态变化会影响是否允许通过当前工作池。
12. 任务列表支持查看各池不同状态数据，也支持暂停、分配、释放等状态变更；状态变更后数据应出现在相应状态页面。
13. 任务统计弹窗展示任务 ID、任务状态、当前工作池、通过的工作池 ID、工作池类型、工作池名称、标注/质检次数。

## 4. Rosetta：项目管理与工作流配置

1. 项目管理承担项目配置、数据集导入、工作流、工作池、操作项、算法、人员分配、任务管理、项目统计和导出等能力。
2. 项目导入数据集时，项目类型和数据集类型需要一致；导入后数据流会自动生成相关联的人工标注和新增版本组件。
3. 只有项目完成后，数据集才会生成新的版本。
4. 项目可配置标注池、检查池、抽查池、完成池，并在人工标注执行侧体现对应入口和流转。
5. 项目管理中的任务列表支持查看、试标、返回、暂停、分配、释放、统计。
6. 项目统计导出支持筛选条件，包括时间段、工作池、统计指标；筛选条件不可重复添加。
7. 导出条件支持 OR 和 AND 类组合逻辑：OR 满足任一条件导出，AND 需同时满足条件导出。
8. 导出弹窗支持取消、关闭和确认导出；确认后浏览器下载文件，文件需可正常打开且内容符合筛选条件。
9. 项目统计、任务统计、抽检包统计是后续统计类需求必须联动回归的历史模块。
10. 项目管理中已有算法配置相关能力，包括添加、编辑、删除算法，以及算法辅助人工标注。

## 5. 操作项配置通用规则

历史用例大量覆盖操作项配置。后续任何涉及标注工具、操作项、属性项、导出字段的需求，都应优先回归以下规则：

1. 操作项有平台展示名称和导出名称；修改后应分别影响标注/查看展示和导出数据字段。
2. 操作项支持自定义颜色和随机颜色；颜色变化应在标注和查看过程中体现。
3. 操作项支持操作提示文字；鼠标悬停右侧对象栏问号时展示提示。
4. 操作项支持最少/最多标注数量；不满足数量限制时点击完成会出现项校验错误。
5. 操作项支持尺寸、点数、精度等校验；不满足最小尺寸、点数或精度时应提示错误并阻止通过。
6. 操作项可能存在父子操作项、子源映射、默认值、属性同步、是否嵌套、选项层级、平级/下级选项增删。
7. 操作项支持不可超出画布配置；关闭后允许超出画布，开启后不允许超出画布。
8. 操作项支持操作教程；开启并输入内容后，标注和查看时点击顶部工具栏项目教程应展示内容。
9. 属性类操作项包括单选、多选、文本输入、数字输入、标签输入、是否选项、姿态估计、多模态文件上传等。
10. 文本输入支持结构化文本渲染类型；数字输入应限制数字；标签模式可输入自定义标签。
11. 连续帧属性同步会影响当前帧、后续帧或所有帧属性值，需要结合视频/点云帧类项目回归。
12. 多模态上传应校验文件类型，例如图片、视频、音频。

## 6. 人工标注多模态能力

1. 图像类项目主要使用 2D 操作项，标注后右侧对象框数量变化，标注结果在配置好的工作池中展示。
2. 视频类项目使用 2D 操作项，支持父子操作项、属性类和其他类操作项，支持底部工具栏倍速播放。
3. 纯文本、富文本、音频类项目均有对应操作项和父子/属性类配置，并应在工作池中正常流转。
4. 点云类项目支持 3D 操作项在 3D 画布标注，2D 操作项在相机视角标注；二者使用场景不可混淆。
5. 点云 3D 标注如果配置子源映射，3D 画布标注可在相机视角显示；相机视角 2D 标注不应反向显示到 3D 画布。
6. 点云类和视频类项目支持按帧播放、暂停、上一帧、下一帧。
7. 点云类项目支持切换不同相机视角。
8. 3D/点云历史高频功能包括点云合并帧、点云聚光灯、范围外点云分割、自车点云分割、单独显示目标物、3D 测量工具、点云高度范围快捷键、一键恢复默认视角、新增线方向反转、重映射属性继承。
9. 标注工具类需求需要同时考虑标注池、检查池、抽查池、完成池的展示与操作一致性。
10. 标注工具新增能力时，需要检查顶部工具栏和右侧功能栏是否互相影响。

## 7. MorningStar：数据集与数据流

1. MorningStar 主要负责数据集、数据版本、数据流、切片、送标、场景标签、数据看板和数据源。
2. 数据集类型覆盖图像、纯文本、富文本、点云、音频、视频。
3. 创建数据集通常需要填写数据集名称和描述，选择数据类型、导入格式和导入方式，导入符合格式的数据文件夹。
4. 创建成功后页面返回数据集列表，创建的数据集展示在列表中。
5. 数据集列表点击数据集信息可进入数据集详情；详情页点击数据预览图可进入数据详情。

11. 版本对比筛选依赖版本关联项目完成状态和标注对象数据，支持标签映射后查询。
12. 数据看板展示数据集数量分布、操作项统计、贡献值排名、热门数据集、最新数据集、标注结果分布、标签分布。

## 9. Phoenix：交付、验收、结算与审批

1. Phoenix 覆盖交付单、客户验收时间、供应商结算单、财务决策、审批流和项目类型。
2. 客户确认时，客户验收时间可能必填；当天和当天之前可提交，晚于系统当天应失败并提示不得选择晚于系统当天时间。
3. 交付记录列表展示客户验收时间；无客户验收时间展示“-”；验收时间不允许二次编辑。
4. 财务决策选择“客户已验收”时，可自动填充交付单填写的验收时间，且不支持修改。
5. 多个交付单场景下，财务决策可能取最晚客户验收时间。
6. 财务决策选择“客户已结算”或“特殊结算款”时，客户验收时间的展示规则与“客户已验收”不同，需要单独覆盖。

8. 审批流可能受金额阈值影响，例如大于 3 万需要 CEO 审批，小于 3 万不经过 CEO。
9. 同一供应商、同一天、同一结算依据的结算单可能合并审批；不同日期、不同供应商或不同依据的合并规则需要分别覆盖。
10. 审批流会触发钉钉提醒项目经理；项目经理评价完成后进入后续审批流程。
11. 项目经理和供应商管理员可能同时看到待评价记录，但评价只允许其中一人完成；项目经理离职时，默认审批可切换到供应商管理员。

# 业务记忆条目：数据快照保存与导出技术实现口径

## 1. 来源

- 技术方案：`/Users/macbook/Downloads/数据快照保存机制与快照结果导出技术方案.docx`
- 对应 PRD：`2026.05.14 新增数据快照保存机制与快照结果导出`
- 解析日期：2026-05-18

## 2. 技术实现核心口径

1. 数据快照使用独立表 `task_annotation_snapshot`，不改变现有 `task_history` 语义。
2. 保存方式采用“前端显式触发 + 后端算法池兜底”。
3. 标注池、检查池、抽查池人工操作由前端在按钮动作后调用保存快照接口。
4. 算法池正常出池由后端在算法池流转逻辑中保存快照。
5. 算法池质检失败退回、回收、错误处理等出池动作需要后端兜底，避免批处理、OpenAPI 或新入口漏调。
6. 保存快照接口为 `POST /rosetta-service/project/task/result/snapshot`。
7. 保存接口请求体包含 `projectId`、`taskId`、`poolId`、`poolType`、`actionType`、`clientActionId`。
8. 前端调用保存接口前，需要确保前置业务接口返回正常；前置接口报错时不调用保存快照接口。
9. 保存快照接口失败时，前端主流程仍按成功处理，后端记录错误日志并发送钉钉通知。
10. 保存接口必须校验任务归属、项目归属和当前用户任务操作权限。

## 3. hash 与去重规则

1. 快照去重只比较 `AnnotationResult.annotations`。
2. 忽略 JSON 空白字符和对象字段顺序。
3. 保留数组顺序，数组顺序不同视为不一致。
4. 不比较 `issues`、`hints`、`metadata`、`quality_message`。
5. 没有 `AnnotationResult` 或 `annotations = null` 时不保存快照。
6. hash 使用规范化后的 annotations 计算 SHA-256。
7. 只与上一条快照 hash 比较，hash 相同跳过，hash 不同保存。
8. 不对 `snapshot_hash` 建唯一索引；任务从 A 到 B 再回 A 时，第二次 A 仍应保存。

## 4. 存储规则

1. MySQL 表 `task_annotation_snapshot` 保存元数据和 `store_url`。
2. SDFS 保存快照负载 JSON，避免大 JSON 进入关系表。
3. SDFS 路径模板为 `rosetta-result/environment/projectId/taskId/taskSnapshot/snapshotId.json`。
4. SDFS 负载包含 `annotationResult` 和 `operators`。
5. `operators` 必须保存触发快照时项目的操作项配置，后续导出历史快照时使用该固化配置。
6. 算法池快照 `operator_id` 为空，不能把机器人或系统用户当作专家。

## 5. 导出规则

1. `ExportDoneTaskRequest` 新增 `includeSnapshots`，默认 `false`。
2. `includeSnapshots = false` 或字段缺省时，保持当前导出格式完全不变。
3. `includeSnapshots = true` 时，必须满足 `poolType = COMPLETE(5)`。
4. `poolId` 必须全部为完成池，沿用完成池导出现有校验。
5. `includeSnapshots = true` 时，每行 JSON 包含 `attachmentType`、`attachment`、`finalResult`、`snapshots`。
6. `finalResult` 使用当前导出逻辑生成，但去掉外层任务原始字段，只保留 `metadata` 和 `extra`。
7. `snapshots` 按 `id ASC` 排序。
8. 快照中的 `operators` 使用 SDFS 负载中固化的 operators，不能使用导出时项目最新操作项。
9. 快照中的 `annotations` 使用保存快照时的 `AnnotationResult.annotations`。
10. `username` 导出时根据 `operator_id` 批量查询，不在快照表固化；用户不存在时返回 `null` 或空字符串。
11. 任务没有快照时，`snapshots` 输出空数组。
12. 分页导出时需要按页批量查询快照，避免按任务 N+1 查询。

## 6. 后续生成测试用例时必须覆盖

1. 保存快照接口参数校验、权限校验、任务和项目归属校验。
2. 前置业务接口失败不调用保存快照接口。
3. 保存快照失败不阻塞主流程，后端日志和钉钉通知必须可观测。
4. 无 AnnotationResult、annotations 为空、对象字段顺序、数组顺序、非 annotations 字段变化、A→B→A 回改。
5. MySQL 元数据、SDFS 路径、SDFS 负载、operators 固化。
6. 算法池后端兜底和幂等，operator_id 为空。
7. includeSnapshots 缺省、false、true、非法非完成池、poolId 混入非完成池。
8. finalResult 结构、snapshots 排序、username 查询、用户不存在、无快照空数组。
9. task_history、工作量统计、流转记录、质检统计和旧导出格式回归。

# 业务记忆条目：数据快照保存机制与快照结果导出

## 1. 来源

- PRD：`/Users/macbook/Downloads/2026.05.14  新增数据快照保存机制与快照结果导出.docx`
- 解析日期：2026-05-18
- 需求版本：V 1.0
- 需求来源：Starbench 项目组

## 2. 核心业务规则

1. 本需求为 Rosetta 标注生产链路新增“数据快照”能力，用于保留专家标注过程中的中间修改结果。
2. 数据快照关注任务生产过程中的 `annotations`，不是只关注完成池最终结果。
3. 标注池点击“完成”即触发快照保存，不以提交成功、任务流转成功或质检算法通过为前提。
4. 检查池点击“合格”或“退回至原操作人/上一操作员”即触发快照保存。
5. 抽查池点击“合格”或“不合格”即触发快照保存，不以后续任务处理为前提。
6. 算法池任务任意流出算法池时触发快照保存，包括正常流向下个池子、质检算法未通过驳回至原操作人、任务回收、错误处理等。
7. 触发保存后，需要比较当前快照 `annotations` 与上一条数据快照 `annotations`；完全一致时不保存新快照，不一致时保存。
8. 上述去重规则只适用于本次新增的数据快照机制，不能影响平台既有历史记录机制或任务流转存储机制。

## 3. 导出规则

1. 项目数据导出在完成池数据导出场景新增“是否携带数据快照（Whether to include data snapshots）”选项。
2. 该选项默认值为“否（No）”，可切换为“是（Yes）”。
3. 只有完成池导出支持携带历史快照，非完成池导出不支持。
4. 选择“否”时，保持当前导出结构不变。
5. 选择“是”时，导出 `.jsonl` 每行任务数据需要包含 `attachmentType`、`attachment`、`finalResult`、`snapshots`。
6. `finalResult` 保存任务最终结果，包含最终 `metadata.preprocessedData.operators`、`metadata.preprocessedData.annotations` 和 `extra`。
7. `snapshots` 保存历史快照数组，每条快照包含 `metadata` 与 `extra`。
8. 每条快照 `extra` 包含 `projectId`、`datasetId`、`poolId`、`taskId`、`status`、`userId`、`username`、`snapshotTime`。
9. 算法池保存的快照中，`userId` 和 `username` 为空。

## 4. 后续生成测试用例时必须补充的覆盖点

1. 标注池点击“完成”但不提交、质检失败或不流转时，仍要校验快照保存。
2. 检查池退回原操作人和退回上一操作员必须分别覆盖。
3. 抽查池“不合格”与后续任务处理要拆开验证保存时机。
4. 算法池正常流出、质检失败驳回、任务回收、错误处理都属于快照保存触发路径。
5. `annotations` 完全一致不保存新快照是核心边界，必须做同池重复和跨池重复两类场景。
6. 选择 `No` 导出必须做旧格式回归，防止新增结构污染老导出。
7. 选择 `Yes` 导出必须校验 JSONL 可解析、字段层级、字段完整性、多任务隔离、特殊字符转义。
8. 权限测试必须同时覆盖页面入口和接口越权，避免无权限用户导出包含历史过程数据。

## 5. 当前仍需产品确认的问题

1. 快照保存失败时是否阻塞任务操作和工作池流转。
2. `annotations` 完全一致的判断是否要求 JSON 严格一致，还是业务语义一致。
3. `snapshots` 导出排序是否固定为保存时间升序。
4. 无历史快照任务选择携带快照导出时，是否固定返回空数组。
5. 是否覆盖所有任务类型，还是仅覆盖 Starbench 当前任务类型。
6. 是否需要新增独立的“携带快照导出”权限。

# 需求记忆：一级属性预标注结果增加确认流程

## 来源

需求文档：`2026.06.03  一级属性预标注结果增加确认流程.docx`

## 核心业务目标

算法池预标注产生的实例，需要标注员手动确认是否有问题后才允许提交题目。历史功能已覆盖实例级预标注结果的确认、修改、删除等触发动作，但没有覆盖一级属性的确认入口，导致预标注算法生成一级属性内容后，标注员无法点击确认，任务因此无法提交。本需求目标是补全一级属性预标注结果的确认与状态流转逻辑，避免一级属性无法确认导致任务提交被阻断。

## 关键业务规则

1. Web 标注工具页面中，预标注算法生成的一级属性需要展示确认状态。
2. 一级属性未确认时，需要阻止任务提交。
3. 一级属性支持通过 Hover 展示确认按钮，并在点击确认后进入已确认状态。
4. 一级属性在未确认或已确认状态下发生内容变更时，需要进入已修改状态。
5. 取消预标注结果中的草稿状态，未确认实例被直接修改后进入已修改状态，不再进入草稿状态。
6. 一级属性不支持触发已删除状态。
7. 实例级既有已确认、已修改、已删除触发动作不能被本次变更破坏。
8. 预标注完成后，如果算法结果包含一级属性，则一级属性后方展示黄色 Icon，表示该一级属性处于未确认状态。
9. 页面存在未确认状态的一级属性时，任务不允许提交。
10. 鼠标 Hover 在未确认状态的一级属性区域时展示确认按钮。
11. 点击未确认一级属性的确认按钮后，该一级属性状态变为已确认，Icon 变为绿色对号。
12. 一级属性处于已确认或已修改等非未确认状态时，Hover 该区域不展示确认按钮。
13. 一级属性处于未确认或已确认状态时，删除、清空、新增、修改一级属性内容，均使该一级属性进入已修改状态。
14. 一级属性进入已修改状态后，Icon 展示为绿色编辑符号。
15. 一级属性无法触发进入已删除状态，即使用户执行删除或清空操作，也不能生成一级属性已删除状态。
16. 未确认实例被直接修改时，包括拖动实例位置，以及添加、修改、删除实例属性、子实例、子属性等操作，实例状态直接变为已修改。
17. 未确认实例被直接修改为已修改后，不需要再次点击确认按钮，也不再因为原未确认状态阻止提交。
18. 草稿状态取消后，页面和接口结果不应再出现草稿状态或草稿图标。
19. 实例级已确认、已修改、已删除等历史状态触发动作需要保持兼容。
20. 一级属性确认或修改后的状态需要在保存、刷新、切换任务后保持一致。
21. 通过前端或接口提交任务时，后端都应校验未确认一级属性，不能只依赖前端按钮逻辑。
22. 需求作用页面为 Rosetta Web 标注工具中的任务处理页面。
23. 第一批使用角色为当前任务标注员或当前任务操作人。
24. 测试数据需要包含算法池预标注生成一级属性的任务，以及只包含实例级预标注结果的回归任务。
25. 若 PRD 未明确提交阻断提示文案，则测试时先按历史实例级提示进行兼容性校验，并把最终文案作为待确认项。

## 测试风险沉淀

1. 只补了前端展示但提交校验仍只识别实例级未确认，导致一级属性未确认被错误提交。
2. 一级属性确认按钮展示区域不准确，Hover 到已确认或已修改属性时仍展示确认按钮。
3. 一级属性修改后仍保留未确认状态，导致用户已经修改但仍无法提交。
4. 删除或清空一级属性错误触发已删除状态，与 PRD 中一级属性无法触发已删除状态冲突。
5. 取消草稿状态时只改一级属性，未确认实例直接修改后仍进入草稿。
6. 多个一级属性只确认其中一条时，系统错误允许提交。
7. 状态只存在前端内存，刷新或切换任务后丢失。
8. 确认接口失败时前端乐观更新为已确认，造成提交数据与真实状态不一致。
9. 后端接口未做未确认一级属性拦截，被接口绕过提交。
10. 实例级已删除、已修改、已确认等历史动作被本次状态枚举调整误伤。
