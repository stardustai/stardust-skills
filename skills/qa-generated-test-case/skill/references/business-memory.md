# 产品业务记忆总表

这份文件只保留“后续新需求仍然大概率会复用”的业务逻辑，不记录一次性的页面文案细节。

## 1. 工作池与角色

1. 当前已知核心工作池包括：标注池、检查池、抽查池。
2. 项目经理负责在工作流配置里维护池子级别的质检算法配置。
3. 当前工作池执行人负责开始标注、完成任务、查看当前池报错和执行提交。
4. 下游工作池执行人可以看到上一池带来的“上游报错”，但该信息只作为背景提示，不额外阻塞本池处理。

## 2. 质检算法强校验 / 弱校验规则

1. 每个质检算法都可以配置校验类型。
2. 新增算法时，校验类型默认是“强校验”。
3. 历史未配置校验类型的算法，当前约定回显为“强校验”。
4. 无论配置多少个算法、校验类型分别是什么，任务完成后都要并行调用所有质检算法。
5. 当前池最终是否允许提交，取决于失败算法中是否包含强校验。
6. 只要失败算法中包含任一强校验，当前池最终状态就必须按“强校验未通过”处理，任务不可提交。
7. 如果失败算法仅包含弱校验，当前池最终状态按“弱校验未通过”处理，任务仍可提交。
8. 当前池重新点击“开始标注”后，再次“完成”必须重新触发全部质检，不能复用旧结果。

## 3. 当前池报错与下游上游报错

1. 当前池中的报错信息用于承接本池质检算法返回的 message。
2. 当任务在当前池因为弱校验失败但仍被直接提交时，需要把弱校验失败信息写入“上游报错”。
3. 多个弱校验算法同时失败时，需要生成多条上游报错记录，而不是聚合为一条。
4. 如果当前池失败算法中包含强校验，则任务不可提交，也不会生成上游报错。
5. 下一个工作池若收到上游报错，应展示“上游报错（Upstream Errors）”入口和提示点。
6. 上游报错标题固定为：`该任务在上游工作池质检算法弱校验未通过，报错信息如下：`
7. 上游报错只在任务流转后的下一个工作池展示一站，不能继续带到更后面的工作池。
8. 上游报错详情过长时需要截断，且支持特殊字符展示。
9. 上游报错写入失败不影响任务流转，但这会导致下一池看不到上游报错，应视为缺陷。

## 4. 通用上游报错与强弱校验风险

1. 强弱混合场景下的优先级判断。
2. 当前池 message 与下游上游报错内容是否一致。
3. 多个弱校验失败时，上游报错是否少写、漏写或条数不对。
4. 强校验失败时，是否被错误放行提交。
5. 弱校验失败时，是否被错误阻塞提交。
6. 上游报错是否在后续两个以上工作池中错误残留。

## 5. 算法池质检失败处理规则

1. 只有在算法池配置了质检算法时，才展示“质检未通过数据处理方式”配置项。
2. 未配置质检算法或清空质检算法时，该配置项必须隐藏。
3. 失败处理方式默认值是“滞留算法池（Staging algorithm pool）”。
4. 失败处理方式支持“滞留算法池”和“驳回至原操作人（Return to original operator）”。
5. 算法池质检失败且选择“滞留算法池”时，任务留在算法池并转为“报错”状态。
6. 算法池质检失败且选择“驳回至原操作人”时，系统尝试退回原操作人；若退回失败，则回落到算法池“报错”状态。
7. 本需求范围内，算法池质检失败默认会返回 message，不考虑空 message 分支。
8. 算法池质检失败返回的每条 message 都应写入独立“上游报错”记录，不写入“报错”详情。
9. 算法池类上游报错标题固定为：`该任务在上游算法池质检未通过被驳回，报错信息如下：`
10. 产品功能只要求任务流转后必须能展示上游报错，具体落库时机不作为本轮产品规则约束。
11. 算法池“报错”状态任务不再执行任何算法，手动触发和自动触发都必须跳过。

## 6. 算法池扩展风险

1. 失败处理方式在未配置质检算法时错误展示，或清空质检算法后未及时隐藏。
2. 默认值不是“滞留算法池”，导致算法池旧流程被意外改变。
3. 驳回原操作人失败时未正确回落到算法池“报错”状态。
4. 多条失败 message 没有逐条生成多条上游报错记录。
5. 算法池失败 message 被写进“报错”而不是“上游报错”。
6. 算法池“报错”状态任务仍被手动或自动触发重复跑算法。
7. 算法池上游报错在下一站缺失、标题错误，或被错误残留到后续多个工作池。

## 7. 数据快照保存机制

1. 数据快照用于保留专家标注过程中的中间修改结果，核心数据是任务快照中的 `annotations`。
2. 标注池点击“完成”即触发快照保存，不依赖后续是否提交、是否流转、质检算法是否通过。
3. 检查池点击“合格”或“退回至原操作人/上一操作员”即触发快照保存。
4. 抽查池点击“合格”或“不合格”即触发快照保存，不依赖后续任务处理是否执行。
5. 算法池任务任意流出算法池时触发快照保存，包括正常流向下个池子、质检失败驳回原操作人、任务回收、错误处理等。
6. 每次触发保存后，需要比较当前快照 `annotations` 与上一条数据快照 `annotations`；完全一致时不保存，不一致时保存。
7. `annotations` 去重规则只适用于新增数据快照机制，不能影响平台已有历史记录、任务流转记录或其他保存历史机制。
8. 算法池保存的快照没有人工操作人，导出时 `userId` 和 `username` 为空。

## 8. 数据快照导出规则

1. 完成池数据导出支持选择是否携带数据快照，字段为“是否携带数据快照（Whether to include data snapshots）”。
2. 该选项默认值为“否（No）”，选择“否”时保持当前导出结构不变。
3. 选择“是（Yes）”时，导出 `.jsonl` 每行任务数据包含 `attachmentType`、`attachment`、`finalResult`、`snapshots`。
4. `finalResult` 表示完成池最终结果，包含最终 `metadata.preprocessedData.operators`、`metadata.preprocessedData.annotations` 和最终 `extra`。
5. `snapshots` 表示历史快照数组，每条快照包含 `metadata` 与 `extra`。
6. 每条快照 `extra` 包含 `projectId`、`datasetId`、`poolId`、`taskId`、`status`、`userId`、`username`、`snapshotTime`。
7. 只有完成池导出支持携带历史快照，非完成池导出不支持。
8. 后续导出类需求都要回归旧格式不变、JSONL可解析、字段完整、多任务隔离、权限越权、特殊字符和大数据量。

## 9. 数据快照技术实现风险

1. 保存快照接口为 `POST /rosetta-service/project/task/result/snapshot`，请求体包含 `projectId`、`taskId`、`poolId`、`poolType`、`actionType`、`clientActionId`。
2. 前端只有在前置业务接口返回正常后才调用保存快照接口；前置业务接口报错时不保存快照。
3. 保存快照接口失败时，前端主流程仍成功，后端记录错误日志并发送钉钉通知。
4. 快照去重只比较 `AnnotationResult.annotations`，忽略JSON空白和对象字段顺序，保留数组顺序，不比较 `issues`、`hints`、`metadata`、`quality_message`。
5. 没有 `AnnotationResult` 或 `annotations = null` 时不保存快照。
6. `snapshot_hash` 不应建唯一索引；任务从A改到B再改回A时，第二次A仍应保存。
7. 快照元数据写入 `task_annotation_snapshot`，快照负载写入SDFS，SDFS负载必须包含 `annotationResult` 和保存时的 `operators`。
8. 算法池出池类动作需要后端兜底保存，并具备幂等能力，算法池快照 `operator_id` 为空。
9. `includeSnapshots = true` 仅允许完成池导出，`includeSnapshots = false` 或缺省必须完全兼容旧格式。
10. `snapshots` 按 `id ASC` 排序；导出时 `username` 根据 `operator_id` 批量查询，用户不存在时返回空值且不应中断导出。
11. 分页导出时应按页批量查询快照，避免按任务逐条查询导致N+1性能问题。

## 10. license管理与私有化授权

1. Phoenix 平台中原 `license生成` 一级入口调整为 `license管理`，管理页通过 `+` 进入 `license管理/license生成`。
2. 从 license 生成页点击标题中的 `license管理` 返回首页时，未提交表单内容不保存。
3. license 生成类型包含 `私有化部署-申请码`、`私有化部署-域名`、`独立标注工具`，新增类型不能破坏独立标注工具旧流程。
4. `私有化部署-申请码` 新增 `申请码`、`项目名称`、`备注`；`申请码` 必填，空提交提示 `请输入邀请码`，`项目名称` 必填，空提交提示 `请输入项目名称`。
5. `私有化部署-域名` 新增 `项目名称`、`备注`；`项目名称` 必填，空提交提示 `请输入项目名称`。
6. license 管理列表支持按项目名称、类别、状态搜索；序号从 1 开始，新增记录递增。
7. 申请码类型记录中申请码取提交内容、域名展示 `-`；域名类型记录中域名取提交内容、申请码展示 `-`。
8. 备注为空时列表展示 `-`，创建人展示提交账号名称，License 内容读取生成结果，复制操作应复制完整 License 内容。
9. 状态展示 `有效` 或 `已过期`；PRD 状态判断口径存在“申请时间”与“有效期”的歧义，后续需求应优先确认到期判断字段。
10. 私有化部署项目在规定期限外仅 Admin 可正常登录，Admin 登录后直接跳转 license 页面。
11. 私有化部署项目 license 页面支持生成申请码、复制申请码、写入 License；License 校验通过后系统授权恢复，普通用户也可继续使用完整系统。
12. 空 License、非法 License 或环境不匹配的 License 不应恢复系统完整授权。

## 11. 任务列表筛选、跨页选中与时间展示

1. 任务列表筛选面板从【筛选】按钮下方弹出，点击【+添加条件】默认出现“当前操作人”。
2. 任务列表筛选维度包括任务ID、标题、是否优先、数据集、当前操作人、当前操作团队、工作池、任务状态等。
3. 多个筛选条件之间默认按 AND 生效；输入筛选值或删除条件后，列表结果和筛选结果数量都要刷新。
4. 筛选结果数量展示在【筛选】按钮右侧，例如 `筛选 368`；无筛选条件时不显示数字，数字应代表筛选命中总量而不是当前页数量。
5. 勾选任务后，【选择全部】按钮右侧展示 `已选择 N 条 ×`；勾选、取消勾选、点击 `×` 清除时，已选中数量和勾选状态要同步更新。
6. 筛选结果数和已选中数相互独立，不能互相覆盖或混算。
7. 任务列表分页大小支持 `500条/页` 和 `1000条/页`，大分页下筛选、勾选、操作列仍需可用。
8. 跨页选中时，不同页面已选任务需要累计统计；返回上一页后已选状态不能丢失。
9. 导出选中任务时，导出范围必须是用户实际选中的任务集合，不能只导出当前页，也不能导出全部筛选结果。
10. 更新时间展示为绝对时间 `YYYY-MM-DD HH:mm:ss`，不再展示 `X 天前`。
11. 任务列表新增【分配时间】列，表示数据第一次流入标注池的时间；从未流入标注池时展示 `-`。
12. 分配时间只记录首次进入标注池时间，后续退回、重新分配、重新流转不能覆盖该值。

## 13. 一级属性预标注确认流程

1. 算法池预标注生成一级属性后，一级属性后方展示黄色 Icon，表示该一级属性处于未确认状态。
2. 页面存在未确认状态的一级属性时，任务不允许提交；该拦截需要同时覆盖前端提交和后端接口提交。
3. 鼠标 Hover 在未确认一级属性区域时展示确认按钮，点击确认后一级属性变为已确认，并展示绿色对号 Icon。
4. 一级属性处于已确认或已修改等非未确认状态时，Hover 不展示确认按钮。
5. 一级属性处于未确认或已确认状态时，删除、清空、新增、修改一级属性内容，均使该一级属性进入已修改状态，并展示绿色编辑 Icon 或按已修改结果参与提交。
6. 一级属性无法触发进入已删除状态；删除或清空一级属性时不能生成一级属性已删除状态。
7. 取消预标注结果的草稿状态后，未确认实例被直接修改时，包括拖动实例位置、添加、修改、删除实例属性、子实例、子属性等操作，实例状态直接变为已修改。
8. 未确认实例被直接修改为已修改后，不需要再次点击确认按钮，也不再因为原未确认状态阻止提交。
9. 实例级已确认、已修改、已删除等历史状态触发动作需要保持兼容，不能被一级属性状态补齐误伤。
10. 一级属性确认或修改后的状态需要在保存、刷新、切换任务后保持一致。

## 14. 一级属性预标注确认流程风险

1. 只补前端展示但后端提交校验仍漏掉一级属性未确认，导致用户绕过页面提交任务。
2. 多个一级属性只处理部分时，系统错误允许提交。
3. 未确认一级属性被修改后仍保留未确认状态，导致用户已经处理但仍无法提交。
4. 删除或清空一级属性错误触发已删除状态，与产品规则冲突。
5. 取消草稿状态时只影响一级属性，没有覆盖未确认实例直接修改的旧草稿链路。
6. 状态只存在前端内存，刷新、保存、切换任务后回退或丢失。
7. 确认接口失败时前端乐观更新为已确认，造成页面状态与真实提交状态不一致。
8. 显隐预标注、撤销、冻结数据、只读查看等历史能力可能破坏一级属性确认状态，需要作为回归雷达。
