# PRD 到测试用例工作流

## 全局约束

每一步都使用以下原则：

1. 只根据 PRD、补充上下文和业务记忆作答，不臆造业务规则。
2. 信息不足时列入 `缺失信息` 或 `待确认问题`。
3. 优先关注角色权限、状态流转、多条件组合、上下游依赖。
4. 业务记忆用于补充历史上下文；当前 PRD 与业务记忆冲突时，以当前 PRD 为准，并指出冲突点。
5. 输出必须结构化、中文、可复用、可进入下一步。

## Step 1：提取业务结构

目标：理解需求，不直接写用例。

必须输出：

1. 需求目标
2. 需求范围内功能点
3. 不在本次范围内的内容
4. 关键角色及其可见范围/可操作范围
5. 主流程
6. 分支流程
7. 已知缺失信息

## Step 2：提取规则、状态和依赖

目标：把业务复杂度转成可测试规则。

必须输出：

1. 角色-权限矩阵
2. 状态流转表
3. 关键校验规则
4. 多条件组合规则
5. 上下游依赖和外部接口影响
6. 高风险漏测点
7. 仍然缺失的信息

## Step 3：生成测试点和覆盖矩阵

目标：先确认“测什么”，再落成用例。

必须输出：

1. 功能测试点列表
2. 业务流程测试点列表
3. 异常场景测试点列表
4. 边界值/组合条件测试点列表
5. 角色权限测试点列表
6. 覆盖矩阵
7. 不确定但建议人工确认的测试点

覆盖矩阵至少包含：覆盖类别、测试点编号、关联功能点、风险说明。

## Step 4：生成最终测试用例

目标：把测试点转成标准 7 列用例。

必须遵守 `references/output-format.md`。

生成要求：

1. 主流程、分支流程、异常场景、边界场景、权限场景都要覆盖。
2. 每条用例只验证一个主要风险点，避免过度合并。
3. 涉及状态流转的用例，预期结果必须写清状态变化。
4. 涉及权限的用例，至少覆盖页面展示、按钮操作、后端校验中的一项或多项。
5. 不要生成基于假设的用例，信息不足内容单独列在 `待确认问题`。

## Step 5：测试负责人自检

不要重写整份结果，只输出：

1. 功能点覆盖缺口
2. 功能场景覆盖缺口
3. 异常场景覆盖缺口
4. 边界值和组合条件覆盖缺口
5. 角色/权限/依赖覆盖缺口
6. 最值得补充的前 10 条用例建议
7. 哪些问题来自 PRD 信息缺失，而不是生成质量差

## 调优原则

1. 先改输出格式，再改描述长度。
2. 先补业务上下文，再补修辞。
3. 一次只改一个步骤，避免不知道质量提升来自哪里。
4. 每次修改后，用同一份 PRD 复测。
