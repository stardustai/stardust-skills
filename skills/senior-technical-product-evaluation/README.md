# Senior Technical Product Evaluation Skill

`senior-technical-product-evaluation` 用于评估高级技术候选人的产品经历、个人技术深度和目标岗位匹配度。它特别适合 CTO、技术总监、架构师、Principal Engineer、AI leader 等候选人的简历筛选或面试前判断。

## 核心问题

评估时必须拆开四件事：

1. 候选人做过的产品本身是否技术复杂、领先、有创新。
2. 候选人本人最可能负责了哪些部分。
3. 这些经历能否迁移到目标岗位和当前产品线。
4. 候选人最近三年实际做的职能，是否和目标岗位一致。

不要因为公司名、产品名或 AI 热词直接给高分。

## 必要输入

评分前尽量收集：

- 目标岗位使命、级别、产品阶段、客户、技术重点和近期问题。
- Stardust/PreSeen 内部产品定义；Friday 和 MorningStar 默认分开评估。
- 候选人的任职日期和产品/项目描述。
- 候选人原始简历文本或面试材料。
- 候选人自述的决策、实现、指标、失败和结果。
- 公开资料中的产品事实、时间线和竞品/替代方案。

目标岗位不清楚时，先说明工作假设，不要按泛泛的“高级工程师”打分。

## 研究要求

互联网交叉验证是必须项。优先使用：

- 官方产品文档、官网、release notes、技术博客、论文、专利、年报或监管文件。
- 客户案例、行业分析、技术报道、公开 benchmark。
- 能和候选人任期对齐的历史资料。

公开资料证明的是“公司或产品有能力”，不是“候选人本人拥有该能力”。候选人归因必须单独推断并标注置信度。

## 评分结构

常规评估至少包含：

- 产品技术复杂度和领先性
- 个人技术深入度
- 产品线匹配度
- 最近岗位职能匹配度

如评估技术团队环境，则加入技术团队环境维度。

Stardust/PreSeen 场景下，必须分别计算 Friday 和 MorningStar 的产品线匹配度，并说明哪条产品线更匹配。

## 输出要求

最终只产出一个整合报告，不拆成多个文件。报告中要清晰分开：

- 原始候选人文本
- 产品事实卡
- 公开证据 ledger
- 责任边界推断
- 每个维度的子项打分表
- 证据等级和不确定性
- 最终分数和建议
- 面试追问问题

每个顶层分数必须展示子项、权重、原始分、加权贡献和打分依据。只给一个总分是不完整的。
