# 风险分级规则

## 高危

满足任一条件可定为高危：
- 可导致未授权访问、越权访问、批量数据泄露、管理员权限绕过。
- 可导致远程命令执行、SQL 注入、SSRF 访问内网、任意文件上传/读取/覆盖。
- 生产或公网环境暴露数据库、Redis、MQ、SSH、管理后台、调试端点。
- 明文硬编码生产凭据、私钥、JWT secret、数据库连接串。
- 核心业务登录、权限、数据导出、删除等关键接口缺少后端控制。

## 中危

满足任一条件可定为中危：
- 安全控制存在但不完整，例如密码策略弱、失败锁定缺失、日志字段不完整。
- 文件上传、CORS、Cookie、安全响应头、输入校验存在明显缺口但暂未证明可直接利用。
- 管理员能力过大、缺少二次确认或审计。
- 敏感数据有过度返回、过度展示、日志泄露风险。
- 发布、备份、回滚流程缺失会放大生产风险。

## 低危

满足任一条件可定为低危：
- 安全配置不规范但影响有限。
- 文档、清单、制度材料不完整。
- 命名、日志、错误处理、提示信息存在改进空间。
- 需要通过运维环境确认才能证明风险可利用。

## 待确认

源码无法证明是否满足要求时使用：
- MFA、证书有效期、线上 TLS 策略、日志保留周期、备份恢复演练。
- 离职账号回收、账号清单、权限审批、发布审批、应急演练记录。
- 云厂商等保、机房物理安全、网络边界设备策略。

## 风险调整

- 面向公网：认证、权限、接口安全、传输安全、暴露面问题上调一级，最高为高危。
- 涉及个人敏感信息、客户数据、支付/财务/审批数据：数据安全和权限问题上调一级。
- 仅测试代码、示例代码、Mock 服务：默认不直接定高危，除非会被打包或部署到生产。
- 有明确补偿控制：可降级，但必须写明补偿控制证据。

## 误报处理

- 不确定时不要夸大。写成 `待确认` 或 `疑似风险`。
- 发现危险 API 但输入不可控时，不直接定为漏洞；说明调用链和约束。
- 发现 `http://` 但只用于本地开发时，记录为低危或待确认。
- 发现缺少文档，不等同于系统必然不安全；标记为材料缺口。
