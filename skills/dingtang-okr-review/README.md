# OKR 评分标准解析

这个 README 是 `dingtang-okr-review` skill 的中文解释版，给管理者和评分执行者阅读。真正执行规则以 `SKILL.md` 为准；本文件用于解释为什么这样评分，以及遇到争议时如何落分。

这里的 `rubric` 可以理解为“评分尺子”：不是看谁写得多、做得苦，而是看 KR 原来承诺的结果有没有真的发生。

## 一句话原则

OKR 评价的是“结果和价值”，不是“动作和工作量”。

如果 KR 承诺的是上线、客户使用、销售转化、效率提升、质量改善、风险下降，那么文档、PPT、PRD、Demo、会议讨论、内部评审都只是中间产物。它们有价值，但不能直接等同于最终结果。

## 产出和结果的区别

`产出` 是做了什么东西，`结果` 是这个东西产生了什么价值。

| 类型 | 例子 | 是否可以直接高分 |
| --- | --- | --- |
| 产出 | PRD、方案、PPT、Demo、会议纪要、代码、测试用例、培训材料 | 只有当 KR 本身承诺这些产出时才可以 |
| 结果 | 上线、真实用户使用、客户接受、销售复用、成交、回款、效率提升、缺陷下降、风险减少 | 可以作为高分依据 |

核心判断：

| KR 原承诺 | 实际只做到 | 判断 |
| --- | --- | --- |
| 写一份方案 | 方案写完并可读 | 可以按产出完成度评分 |
| 方案支持成交 | 只有方案，没有客户使用或商机推进 | 不能高分 |
| 完成技术 Demo | Demo 可运行并有验收 | 可以按 Demo 标准评分 |
| 上线并提升效率 | 只有 Demo 或代码，没上线、没效率数据 | 不能高分 |

## 为什么“文档、Demo、会议讨论”通常只能在 40-60 分

这条规则不是否定文档、Demo 或会议。它们完成了思考、设计、技术验证、可行性验证，所以不是 0 分。

但如果 KR 的核心目标是让客户获得价值、让产品被使用、让销售产生机会、让效率变高，那么只有中间产物还没有证明目标实现。

参考分段：

| 实际完成情况 | 常见分数 |
| --- | ---: |
| 只有想法，没有产物 | 0-20 |
| 有文档、方案、PRD、设计稿 | 30-45 |
| 有 Demo，能证明基本可行 | 45-55 |
| 有 Demo，且接近上线或试点 | 55-65 |
| 已上线、被使用、被复用、有客户或数据验证 | 70-100 |

注意：这不是所有 KR 的统一封顶。工程类 KR 如果原本只承诺“完成 SDK 封装”，代码、测试、文档本身就可能是结果。业务类 KR 如果承诺“支持成交”，方案就只是前置动作。

## 评分档位

| 分数 | 判定 |
| ---: | --- |
| 100 | 原 KR 完全达成，有独立证据，且产生预期价值 |
| 80 | 大部分达成，核心结果成立，只有小缺口 |
| 60 | 部分达成，但核心结果不完整或效果证据弱 |
| 40 | 有明显动作或中间产物，但重要结果缺失 |
| 20 | 只有少量迹象，无法证明有效完成 |
| 0 | 无证据、无尝试、证据矛盾，或做了与 KR 无关的替代动作 |

## 通用评分例子

KR：Q2 完成 3 个客户认可的 PoC。

| 实际证据 | 建议评分 |
| --- | ---: |
| 3 个 PoC 都有客户验收或明确认可 | 100 |
| 2 个已验收，1 个进入最终验收 | 80 左右 |
| 1 个验收，2 个只有 Demo | 60 左右 |
| 只有内部 Demo，没有客户认可 | 40 左右 |
| 只有会议记录或口头讨论 | 20 左右 |
| 没有相关证据 | 0 |

## 销售类 KR 样例

销售类 KR 的关键是客户、商机、合同、回款、转化，不是拜访次数或材料数量本身。

KR：发现 20 个有效客户需求，并转化 5 个合格商机。

| 实际证据 | 判断 |
| --- | --- |
| 只有拜访记录和需求列表，没有客户确认或 CRM 商机 | 通常 40-60 |
| 有客户确认的需求，但只有 2 个合格商机 | 按数量和质量部分给分 |
| 有 20 个客户确认需求，5 个进入 CRM 合格商机 | 可到 80-100 |
| 自评写“客户很感兴趣”，但无客户、商机、下一步证据 | 低分 |

高分证据包括：客户确认、CRM 商机、采购链路、下一步计划、PoC 范围、合同、回款、销售漏斗推进。

## 售前和解决方案类 KR 样例

解决方案类 KR 要先判断承诺的结果属于哪一类：

| 承诺结果 | 归类 |
| --- | --- |
| 客户方案、打单、PoC、复用率 | 业务 / GTM |
| 标准 offer、产品包、产品架构 | 产品 |
| 工具链、组件、部署脚本、自动化交付 | 工程 |

KR：沉淀 3 套行业解决方案，并达到 50% 售前复用率。

| 实际证据 | 判断 |
| --- | --- |
| 只有 3 个 PPT | 通常 40-60 |
| 有方案，销售/售前在 2 个客户中使用 | 可以部分给分 |
| 有复用记录、客户反馈、版本迭代、售前采用率 | 可到 80+ |
| 有方案但客户认为太 low，售前不用 | 分数应明显下调 |

高分证据包括：客户使用、售前复用、复制后的方案版本、PoC 推进、销售反馈、客户反馈、标准方案库使用记录。

## Marketing 类 KR 样例

Marketing 类 KR 不能只看发布了多少内容，要看内容是否带来线索、询盘、品牌触达、销售复用或产品认知提升。

KR：发布 8 篇行业内容，并产生 30 个有效线索。

| 实际证据 | 判断 |
| --- | --- |
| 8 篇文章已发布，但无阅读、询盘、线索数据 | 通常 40-60 |
| 有阅读量和转发，但没有有效线索 | 中低分 |
| 有 30 个有效线索，并交给销售跟进 | 可到 80+ |
| 有内容被销售用于打单或客户教育 | 可作为加分证据 |

高分证据包括：内容链接、阅读/转发数据、表单线索、销售接收记录、客户询盘、商机转化、销售/售前复用。

## 产品类 KR 样例

产品类 KR 的核心是产品价值是否被验证。PRD、产品架构、路线图、Demo 可以证明思考和准备，但不能自动证明产品成功。

KR：上线标准 offer，并让销售和售前稳定复用。

| 实际证据 | 判断 |
| --- | --- |
| 有 PRD、产品架构图、Demo | 通常 40-60 |
| 已上线，但没有使用数据或复用证据 | 仍然不能高分 |
| 已上线，有销售包、售前复用、客户反馈 | 可到 80+ |
| 已上线并带来商机、试点、转化或使用增长 | 可接近满分 |

高分证据包括：版本链接、发布记录、使用数据、用户反馈、销售/售前复用、客户验证、PoC 转化、研发/测试验收、问题闭环。

## 产品机制类 KR 样例

产品机制包括 VOC、P0 需求规则、产品优先级机制、产品培训、统一口径等。机制被宣布不等于机制有效。

KR：建立 VOC/P0 规则，并让低质量需求比例下降 30%。

| 实际证据 | 判断 |
| --- | --- |
| 规则文档存在 | 通常不超过 50-55 |
| 规则用过一次，但无统计 | 通常仍然偏低 |
| 有需求池、拦截记录、低质量需求比例前后对比 | 可高分 |
| 有销售/售前/研发共同采用并定期复盘 | 可作为强证据 |

## 工程类 KR 样例

工程类 KR 要先看原承诺。如果原承诺是技术产物，技术产物本身就是结果；如果原承诺是效率、稳定性或业务效果，就必须看效果指标。

KR：完成 SDK 封装，并提供 5 个集成样例。

| 实际证据 | 判断 |
| --- | --- |
| 有代码、PR、包文档、测试报告、5 个样例 | 可高分 |
| 有代码但无测试、无文档、无样例 | 部分给分 |
| 只有设计文档，没有实现 | 通常 40-60 |

KR：AI 工具让交付效率提升 30%。

| 实际证据 | 判断 |
| --- | --- |
| 工具代码完成，但没人使用 | 不能高分 |
| 有团队使用，但没有前后效率数据 | 中等偏低 |
| 有使用日志、交付周期前后对比、任务吞吐提升 | 可高分 |

高分证据包括：代码路径、合并 PR、测试报告、发布记录、使用日志、性能指标、故障下降、周期缩短、团队采用记录。

## 算法类 KR 样例

算法类 KR 不能只看模型训练或实验完成，要看 KR 承诺的是算法指标、项目落地，还是业务替代。

KR：模型准确率提升 10%，并在真实项目中使用。

| 实际证据 | 判断 |
| --- | --- |
| 有实验结果，但无评测集说明 | 证据弱 |
| 有评测集、准确率提升，但未进项目 | 完成算法部分，落地部分扣分 |
| 有指标提升，并在真实项目中节省人工或提升质量 | 可高分 |

高分证据包括：评测集、baseline、指标对比、项目接入记录、人工节省、质量提升、客户或交付验收。

## 文档类产出如何评分

如果 KR 原文明确承诺“产出文档、方案、SOP、报告、PRD、Demo、产品全景图、技术架构图、产品矩阵说明、投资人讲法”，这些都是可验证产出，必须找到并读过。

KR：产出产品全景图、技术架构图、产品矩阵说明、投资人讲法和 Demo。

| 实际证据 | 判断 |
| --- | --- |
| 只找到 Demo | 只给 Demo 对应部分 |
| 找到全部文档，但没人使用 | 文档完成，但落地价值打折 |
| 找到全部文档，并被销售、投资人、产品评审使用 | 可以高分 |
| 找不到文档，只在进度里写“已完成” | 视为证据不足 |

## 多点 KR 必须逐条评价

如果 KR 写了 `1、2、3`，评分时也必须按 `1、2、3` 评论，不能合并成一句“有进展”。

例子：

KR：

```text
LLM 数据领先性：
1、EDBench 确定生产目标数量，完成立项，并完成 Q2 生产目标；
2、发布 StarBench VLM Reasoning 1.0 版 3000 条；
3、产出可用于打单的 Benchmark 差异化分析报告，并有靠认知成单的案例。
```

正确评价方式：

```text
1）检查 EDBench 是否立项、目标数量是否明确、Q2 生产数量是否完成。
2）检查 StarBench 是否发布、版本是否为 1.0、数量是否达到 3000 条。
3）检查 Benchmark 报告是否存在、是否可用于销售打单、是否有成单案例。
总体：按三项完成度、证据强度和业务价值综合给分。
```

## 降级执行和替代交付

如果原 KR 承诺 A，实际做了更小或不同的 B，不是自动 0 分，也不能当成 A 完成。要看 B 对原目标还有多少价值。

例子：

| 原 KR | 实际完成 | 判断 |
| --- | --- | --- |
| 产品内嵌 NPS，并用于产品迭代 | 临时表单收集 20 条反馈 | 有反馈价值，但产品闭环未实现，明显打折 |
| 客户自助使用系统 | 内部同事手工帮客户跑 | 有客户服务价值，但不是自助使用 |
| 自动化测试覆盖核心流程 | 人工补测核心流程 | 有质量保障动作，但自动化目标未达成 |

## 指标类 KR

指标类 KR 必须看实际数字和影响。

| KR | 高分需要 |
| --- | --- |
| NPS 上线并收集 100 份反馈 | 产品内嵌或明确入口、反馈数量、分析结论、迭代动作 |
| 使用频次提升 30% | 口径、基线、当前值、统计周期 |
| ROI 提升 | 成本、收益、计算方式 |
| 交付周期下降 | 前后周期数据、样本范围 |
| 质量提升 | 缺陷率、返工率、告警、验收质量 |

如果指标没有上线，只做了临时表单或人工统计，要明确写“执行方式降级”，并按真实价值折扣。

## 活动类 KR

如果 KR 只承诺活动本身，活动完成可以得分；如果 KR 承诺活动带来业务结果，就要看结果证据。

| KR | 判断 |
| --- | --- |
| 完成 4 次客户 workshop | 4 次活动记录可以作为核心证据 |
| 完成 4 次 workshop，并推动 3 个商机 | 活动记录不够，还要商机证据 |
| 完成 6 次产品培训，并统一销售话术 | 培训材料不够，还要销售话术复用或考核结果 |

## 逾期扣分

先按结果质量给基础分，再按逾期时间扣分。逾期不是直接否定结果，但会反映执行兑现能力，也可能让业务窗口失效。

| 延迟情况 | 常见处理 |
| --- | --- |
| 晚 2 周以内 | 轻微扣分，通常不超过 10 分 |
| 晚 1 个月左右 | 明显扣分，通常最高 70-80 |
| 晚 2 个月左右 | 即使完成，通常最高 60 |
| 晚超过 2 个月 | 通常最高 40-50 |
| 错过客户、收入、上线、管理决策窗口 | 按实际业务损失继续扣分 |

## 自评进度如何使用

自评进度是线索，不是最终分数。

| 自评情况 | 处理方式 |
| --- | --- |
| 自评 100%，但只有方案 | 看 KR 是否只承诺方案；如果承诺客户接受或成交，不能高分 |
| 自评 80%，但给出 CRM 明确数字 | 若后台不可访问，可暂信数字并记录审计缺口 |
| 自评很低，但独立证据显示结果完成 | 可以按证据提高评分 |
| 自评描述很笼统 | 标记证据不足，要求补充具体结果 |

## 评分时的推荐表达

好的评价要说明“哪里完成、哪里没完成、为什么是这个分数”。

推荐写法：

```text
系统进度显示已完成方案和 Demo，独立检索也找到方案文档；但 KR 原承诺是客户复用和商机推进，目前未看到客户接受、售前复用、PoC 推进或 CRM 商机证据。因此按中间产物完成、业务结果未验证评分。
```

不推荐写法：

```text
完成较好，建议继续加强。
```

## 最常见错误

| 错误 | 正确做法 |
| --- | --- |
| 把工作量当结果 | 回到 KR 原文，看承诺的结果 |
| 把自评当证据 | 自评只能作为线索 |
| 把文档存在当业务成功 | 继续查使用、复用、客户、数据 |
| 多点 KR 只写一句总评 | 逐条评价每个子项 |
| 替代交付直接当完成 | 写清楚原承诺和实际交付的价值差 |
| 逾期完成不扣分 | 按延迟时间和业务窗口扣分 |

## 最终判断问题

每个 KR 最后都要回答三个问题：

1. 原 KR 到底承诺了什么结果？
2. 现有证据能证明这个结果发生了吗？
3. 如果没有，只完成了哪些中间产物，这些中间产物还剩多少价值？

这三个问题回答清楚，分数通常就不会偏。

## 可复用审计脚本

Skill 内置三类只读脚本，用于把“临时查接口、人工比 Excel、肉眼看在线表”变成可重复执行的验收流程。

| 脚本 | 用途 | 是否写系统 |
| --- | --- | --- |
| `dingteam_okr_api.py` | 读取评分档案、评分批次和单个目标评分明细 | 否 |
| `dingteam_okr_audit.py` | 保存基线、比较前后差异、扫描空分/零分/重复考核 | 否 |
| `dingteam_okr_consistency_audit.py` | 对比系统、预期清单、Excel 和钉钉在线表 | 否 |

### 1. 评分前保存基线

```bash
SKILL_DIR=/Users/derek/.agents/skills/dingtang-okr-review
python3 "$SKILL_DIR/scripts/dingteam_okr_audit.py" snapshot \
  --okr-id "$OKR_ID" \
  --output baseline.json
```

输出会带采集时间、季度 ID、完整评分档案和文件 SHA-256。基线应在批量写入前立即生成，期间不能同时运行其他改分任务。

### 2. 评分后检查意外变化

```bash
python3 "$SKILL_DIR/scripts/dingteam_okr_audit.py" compare \
  --okr-id "$OKR_ID" \
  --baseline baseline.json \
  --output comparison.json
```

比较结果分为：

- `added`：新增目标。
- `removed`：目标被删除。
- `changed`：分数、进度、目标名、人员或部门发生变化。
- `hasDifferences`：是否存在任意差异。

例子：计划只更新 Mina 的 7 个管理评价，但比较结果还出现 ET 的目标被删除，必须停止后续写入并调查，不能把删除当成正常清理。

### 3. 扫描空分、零分和重复考核

```bash
python3 "$SKILL_DIR/scripts/dingteam_okr_audit.py" analyze \
  --archive baseline.json \
  --category leadership=领导力 \
  --category culture=文化价值观
```

分类条件由命令行传入，不把“领导力、文化价值观”写死在程序里，因此也可以检查其他类型。`nullScores` 是没有评分，`zeroScores` 是明确打了 0 分，两者业务含义不同。`duplicates` 只表示同一人员同一类型有多个目标，需要人工判断，不允许脚本自动删除。

### 4. 验证系统、Excel 和在线表一致

```bash
python3 "$SKILL_DIR/scripts/dingteam_okr_consistency_audit.py" \
  --okr-id "$OKR_ID" \
  --manifest final_manifest.json \
  --workbook review.xlsx \
  --online-node "$DINGTALK_SHEET_NODE" \
  --output consistency.json
```

`manifest` 是本次评分的预期状态，每个人至少包含：

```json
{
  "person": "Mina",
  "systemEvaluators": ["Derek Zen"],
  "rows": [
    {
      "objectiveId": "objective-id",
      "krId": "kr-id",
      "kr": "战略能力",
      "score": 75,
      "scoreDesc": "系统中应保存的完整评价文本"
    }
  ]
}
```

脚本逐项检查：

1. 实时系统中目标和 KR 存在。
2. 完成人必须与 `systemEvaluators` 一致，状态必须是 `complete`。
3. 系统 0-1 分值换算为 0-100 后与 manifest 一致。
4. 系统评价文本与 `scoreDesc` 完全一致，避免只写入半段评论。
5. Excel 对应人员 tab 和 KR 行存在，分数一致。
6. 如果提供 `--online-node`，钉钉在线表回读值与 Excel 一致。

任一项不一致，程序返回非零退出码并在 `errors` 中给出人员、KR、位置和差异。只有 `errorCount: 0` 才能对外宣称评分、Excel 和在线文档已经同步完成。

### 安全边界

- 脚本只允许白名单读取接口，不含改分和删除接口。
- 认证复用 Skill 的专用 headless profile，不读取用户普通 Chrome 的 cookie，不打印 token 或请求头。
- 删除重复目标、重打分和覆盖评价属于写操作，必须另行获得明确授权。
- 线上验收必须实时读取系统，不用历史 JSON 冒充当前状态。
