# 数据库与迁移标准

## 数据库选型

- 在没有批准例外时，关系型数据库采用 PostgreSQL。先评估组织托管数据库，
  高可用、备份、补丁和灾备能力不能由单个 StatefulSet 替代。
- 用户明确指定其他数据库时，记录兼容性、运维能力、许可、备份和 HA 理由，
  不得偷偷替换用户已有数据库。

## 迁移设计

- 把迁移脚本、执行配置和恢复说明放在 `deploy/db-migrations/`；禁止在应用启动
  命令中执行 Schema Migration 或初始化数据。
- Kubernetes 默认使用单独触发的有限期 Job，不默认使用每个 Pod 都会运行的
  Init Container 或长期 Sidecar。Job 设置数据库互斥锁、超时、重试上限、
  ServiceAccount、资源限制、安全上下文和成功/失败日志。
- Compose 使用独立 `docker-compose.migration.yml` 或 profile 隔离的一次性服务，
  在审批窗口执行并以退出码阻断流水线，完成后删除容器。
- 迁移使用 expand/contract：先添加兼容结构，发布双读/双写或兼容代码，回填并
  验证，再在后续窗口删除旧结构。避免一次发布完成不可逆破坏。
- 对每次变更定义经过验证的恢复机制。可逆变更可以使用 Down Migration；数据
  删除或转换使用前向修复、备份恢复、快照或时间点恢复。不得伪造无法安全执行
  的回滚脚本。

## 发布门禁

- 在代表性非生产数据上验证升级、重复执行、并发互斥、部分失败、超时、应用
  向前/向后兼容及恢复路径。
- 执行前确认备份或 PITR 可用并记录恢复点；明确负责人、观察指标、停止条件、
  最大锁表时间和恢复耗时。
- 将迁移校验和、执行身份、开始/结束时间、结果和审批号写入审计系统，不在日志
  中暴露数据内容或凭据。
