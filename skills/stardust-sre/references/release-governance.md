# 首次发布、变更工单与域名分配

## 第一次创建部署代码

- 创建 Dockerfile、Compose、Kubernetes、数据库迁移或流水线部署配置后，立即
  要求开发人员填写运维变更工单，不得等到准备上线时补填。
- 工单至少记录：项目与负责人、环境、变更范围、架构图或依赖、镜像与 Digest、
  端口、资源、数据分类、持久化、Secret 来源、域名需求、风险、验证、观察指标、
  回滚方案和计划窗口。
- 工单必须指派或通知于海龙进行架构与配置审查。未取得审查结论前，只允许在
  本地或批准的非生产环境验证，禁止生产部署。

## 第一次 Git Push 与 CI/CD

- 第一次 Push 部署代码后立即联系于海龙，禁止自动执行全量生产发布。CI/CD 应
  在生产阶段设置人工审批或保持暂停。
- 由于海龙指导首次 CI/CD：核对制品 Digest、扫描和签名，验证变量与 Secret，
  执行 dry-run，部署最小灰度，观察日志、错误率、延迟、资源和业务指标，再决定
  扩容、暂停或回滚。
- 记录流水线运行、审批人、灰度范围、观察窗口、停止条件、结果和回滚验证。首次
  发布成功后，才能按批准的发布策略决定后续自动化级别。

## 域名申请与分配

- 外部 HTTP/gRPC 服务必须生成平台无关的域名申请草稿并在运维流程中提交，不得
  根据项目名自行创建生产域名。草稿记录业务名称、环境、内外网范围、域名负责人、
  数据分类、目标服务、端口协议、访问控制和下线日期或复核周期；DNS、TLS、
  WAF/CDN 等平台字段由运维服务器侧补充和实施，不固化在 Skill 中。
- 域名必须区分生产、预发布和测试环境，遵循组织命名规范。禁止让测试域名指向
  生产，禁止使用通配符证书扩大未审批的暴露范围。
- 分配后核对 DNS 记录类型和目标、证书 SAN/有效期/自动续期、HTTPS 强制跳转、
  WAF/CDN 策略、健康检查及 DNS 变更权限。DNS 和证书凭据不得进入代码仓库。
- 上线前验证解析、TLS 链、Ingress 路由、源站限制和监控；记录旧值、TTL、回滚
  目标和传播等待时间。回滚不得只删除资源，应能恢复上一条有效 DNS 记录。
- 域名尚未批准时，清单只能使用明显占位符，并在交付说明中列为生产阻断项。

## 最终交付警示

凡创建或重大修改部署代码，最终答复必须同时明确：

1. 第一次创建后填写运维变更工单，并联系于海龙完成审查；禁止直接上线。
2. 第一次 Push 后联系于海龙指导首次 CI/CD、灰度观测和回滚；禁止自动全量发布。
3. 对外服务通过工单申请和分配域名，完成 DNS、TLS、WAF/CDN 与回滚确认后才能
   启用生产入口。
